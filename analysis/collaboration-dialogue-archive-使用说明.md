# collaboration-dialogue-archive Skill 使用说明

> **一句话定位**：把你和合作者、投资人、业务伙伴、产品/技术团队、客户或行业专家的对话，整理成可追溯、可检索、可进入 AI 原生商业平台战略设计的结构化档案。

---

## 快速入口

| 我想做什么 | 直接说 |
|---|---|
| 记录一段对话 | "帮我归档这次和某某的对话" |
| 整理访谈内容 | "整理这次访谈，提炼关键观点和后续动作" |
| 从聊天里提炼战略输入 | "把这段对话沉淀成 AI 原生平台的战略输入" |
| 总结共识和分歧 | "汇总这几次对话里的共识、分歧和待验证假设" |
| 提炼架构影响 | "从这次对话里提炼对业务层、应用层、技术层、数据层的影响" |
| 只做普通纪要 | "只整理成普通会议纪要，不需要 AI 原生平台分析" |

---

## 适合记录哪些对话

这个 skill 适合记录对公司方向、产品形态、技术架构、数据资产、商业模式或组织协同有长期价值的对话。

| 对话对象 | 适合沉淀的内容 |
|---|---|
| CEO / 创始人 | 愿景、战略判断、资源投入、关键取舍 |
| 投资人 / 董事 | 融资叙事、增长逻辑、风险判断、关键指标 |
| CPO / 产品负责人 | 用户场景、产品边界、优先级、体验假设 |
| CTO / 技术负责人 | 架构路线、模型选型、系统约束、工程风险 |
| 业务负责人 | 市场机会、客户需求、运营约束、商业闭环 |
| 客户 / 种子用户 | 真实痛点、工作流、购买意愿、信任门槛 |
| 行业专家 | 行业规则、隐性知识、监管要求、竞争格局 |
| 合作伙伴 / 生态方 | 集成路径、数据交换、利益分配、联合交付 |

## 基本用法

### 方式一：直接给原始对话

把聊天记录、会议摘录、语音转写或你自己的回忆贴给 Codex，然后说：

```text
用 collaboration-dialogue-archive 归档这次对话。
对话对象是某某，角色是某某。
重点帮我提炼：关键观点、分歧、隐含假设、对 AI 原生商业平台的影响。
```

### 方式二：边聊边补充

如果材料不完整，可以先说：

```text
我想记录一次和投资人的对话。你先按归档结构问我关键信息。
```

Codex 会先补齐日期、对象、角色、主题、对话类型，再进入归档。

### 方式三：多次对话综合

当你已经有多份归档，可以说：

```text
把最近三次和 CEO、产品负责人、客户的对话做一次综合分析。
重点看共识、分歧、风险和下一步需要验证的假设。
```

---

## 标准输出结构

默认输出是一份结构化对话档案：

| 模块 | 内容 |
|---|---|
| 基本信息 | 日期、对象、角色、对话类型、关联主题、重要程度 |
| 一句话摘要 | 这次对话最重要的战略价值 |
| 背景 | 为什么发生这次对话，对方站在什么位置 |
| 关键观点 | 对方观点、你的判断、隐含假设 |
| 分歧与张力 | 尚未达成一致的判断、利益差异、认知差异 |
| AI 原生平台启发 | 分别提炼业务层、应用层、技术层、数据层影响 |
| 决策影响 | 已影响的决策、可能影响的决策、需要暂缓的判断 |
| 风险与反证 | 不能过早相信的判断、需要验证的反向证据 |
| 后续行动 | 行动项、负责人、截止时间 |
| 可检索标签 | 方便之后按主题、角色和战略价值检索 |

## AI 原生平台四层提炼

这个 skill 的核心价值，是把普通对话转化为 AI 原生商业平台的设计输入。

| 层 | 关注问题 |
|---|---|
| Business Layer | 这次对话是否影响市场定位、商业模式、增长逻辑、竞争壁垒？ |
| Application Layer | 是否影响产品形态、用户体验、人机协作方式、工作流重构？ |
| Technical Layer | 是否影响模型、RAG、Agent、知识图谱、系统架构或集成方式？ |
| Data Layer | 是否暴露新的数据资产、数据闭环、知识结构或反馈飞轮机会？ |

如果某一层没有明显影响，应该标记为"暂无明确影响"，不要硬编。

## 推荐提示词

### 归档单次对话

```text
用 collaboration-dialogue-archive 帮我归档以下对话。
对话对象：某某，角色：某某。
请重点保留对方原意，区分事实、观点、我的判断和你的推断。
最后提炼对 AI 原生商业平台四层架构的启发。
```

### 整理语音转写

```text
下面是一次语音转写，内容比较散。
请用 collaboration-dialogue-archive 整理成结构化对话档案。
不要过度美化，保留分歧、不确定性和待验证假设。
```

### 提炼行动项

```text
请从这次对话中提炼后续行动项。
每个行动项尽量包含：动作、负责人、截止时间、依赖条件。
如果负责人或时间没有提到，请标记为未注明。
```

### 多方综合分析

```text
请用 collaboration-dialogue-archive 综合以下几份对话归档。
输出：总体结论、已形成共识、明显分歧、反复出现的主题、
对 AI 原生平台四层架构的影响、需要继续确认的问题。
```

## 标签与分类

归档后建议至少打三类标签：

| 标签类型 | 示例 |
|---|---|
| 人物/角色 | `#CEO` `#投资人反馈` `#客户访谈` `#合作伙伴` |
| 战略主题 | `#商业模式` `#产品战略` `#技术架构` `#数据飞轮` |
| 处理状态 | `#待跟进` `#已形成决策` `#需再次确认` `#待验证假设` |

常用价值分类：

- `战略判断`
- `产品洞察`
- `技术架构`
- `组织协同`
- `市场验证`
- `风险预警`
- `待验证假设`

## 使用建议

1. **尽量保留原话**  
   尤其是对方的强判断、反对意见、犹豫和条件句。这些往往比顺滑总结更有价值。

2. **不要把所有内容都变成结论**  
   对话里有些内容只是线索，应该进入"待验证假设"，而不是直接变成决策。

3. **每次都区分事实、观点和推断**  
   事实可以进入知识库，观点可以进入判断库，推断需要标注来源。

4. **多次对话后再做综合**  
   单次对话适合归档，多次对话适合提炼共识、分歧和战略方向。

5. **和 ai-native-strategy 配合使用**  
   当对话归档积累到 3-5 份后，可以把综合分析作为 `ai-native-strategy` 的输入，用于正式架构设计。

## 示例

### 输入

```text
帮我归档这次和供应链合作伙伴的对话。
他说现在最大的问题不是找货，而是判断哪些需求是真实需求。
他们愿意提供历史订单和库存数据，但担心数据给出去以后失去议价权。
我判断这里可能有一个 AI Agent 帮他们做需求真实性判断的机会。
```

### 输出摘要示例

```markdown
## 一句话摘要

这次对话显示，供应链合作伙伴的核心痛点不是信息获取，而是需求真实性判断；
这可能成为 AI 原生平台在供需匹配场景中的关键切入点。

## 隐含假设

- 合作伙伴愿意共享数据，但前提是平台不能削弱其议价权。
- 需求真实性判断可以通过历史订单、库存、客户行为和上下游反馈进行建模。
- AI Agent 的价值不只是推荐货源，而是帮助判断"哪些需求值得响应"。

## 对 AI 原生商业平台的启发

### Business Layer

- 平台的价值主张可以从"撮合供需"升级为"提高需求判断质量"。

### Application Layer

- 可设计面向合作伙伴的需求可信度评分、异常需求提醒、响应建议。

### Technical Layer

- 可能需要 Agent + RAG 结合历史订单、库存和客户画像进行判断。

### Data Layer

- 历史订单、库存变化、询盘转化率可以构成需求真实性判断的数据飞轮。
```

---

## 不适合的用法

- 只想做极简会议纪要，且不需要长期归档。
- 对话完全没有战略、产品、技术、数据或组织价值。
- 需要逐字逐句法律留痕，这类场景应使用正式录音、转写和审计工具。
- 涉及高度敏感信息但没有明确脱敏要求。

---

## 推荐归档节奏

| 节奏 | 做什么 |
|---|---|
| 每次重要对话后 | 生成单次对话归档 |
| 每周 | 汇总本周关键对话、行动项和风险 |
| 每月 | 综合多方观点，更新战略假设和待验证问题 |
| 融资/重大决策前 | 汇总投资人、客户、团队和合作伙伴观点，形成决策输入 |
