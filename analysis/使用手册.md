# ai-native-strategy Skill 使用手册

> **一句话定位**：帮你做 AI 原生产品的技术战略与架构设计，从利益相关者调研，到四层架构分析，到投资人/技术团队双版本文档输出，并支持版本比对。

---

## 快速入口

| 我想做什么 | 直接说 |
|---|---|
| 开始做架构设计 | "帮我做 AI 原生产品的架构分析" |
| 记录某人的访谈 | "记录 CEO 的访谈内容" |
| 比对上一版分析 | 上传旧报告 + "做第二版，对比上一版" |
| 生成战略报告 | 完成层分析后说 "生成战略报告" |
| 生成执行级文档 | 某层分析后说 "生成执行级补充文档" |
| 生成 PPT | 有战略报告后说 "生成 PPT" |

## 三种模式

### 模式 A — 利益相关者调研

**触发**：提到"记录访谈"、"CEO 想法"、"投资人反馈"、"用户调研"等

**做什么**：
- 逐角色收集输入（6 种角色模板，见下表）
- 每个角色生成一份结构化文档
- 有多份记录后可生成**多方综合分析**

**支持的角色**：

| 角色 | 关注点 |
|---|---|
| CEO / 创始人 | 愿景、市场判断、资源投入 |
| 投资人 / VC | 回报逻辑、关键指标、风险判断 |
| CTO / 技术负责人 | 技术栈、团队能力、技术约束 |
| CPO / 产品负责人 | 用户研究、产品愿景、功能优先级 |
| 域专家 / 行业顾问 | 行业知识、监管、竞争格局 |
| 种子用户 | 当前痛点、工作流、付费意愿 |

**输出文件**：
- `stakeholder-<角色>-<姓名>.md`（每人一份）
- `stakeholder-synthesis.md`（多方综合，按需生成）

---

### 模式 B — 架构设计（主流程）

**触发**：提到做架构分析、技术战略、AI 原生产品设计等

**完整步骤**：

```
Phase 1：诊断
├── Step 1-8：产品核心诊断（逐步引导，每次一个问题）
│   ├── Step 1：产品核心与目标用户
│   ├── Step 2：AI 原生测试（去掉 AI 还能存在吗？）
│   ├── Step 3：用户对 AI 的接受度
│   ├── Step 4：私有数据资产
│   ├── Step 5：领域知识复杂度
│   ├── Step 6：团队 AI/ML 能力
│   ├── Step 7：硬性约束（成本/延迟/合规）
│   └── Step 8：时间线与野心
└── Step 9：创始团队资产诊断（有既有业务时运行）
    ├── 9A：市场地位与客户资产
    ├── 9B：供应链与上游关系
    ├── 9C：现有数据资产
    ├── 9D：技术资产积累
    ├── 9E：团队执行力
    └── 9F：早期验证信号

→ 生成产品摘要 + 创始团队资产摘要，确认后继续

Phase 2：四层架构分析（可自选层）
├── 业务层：变革论题、商业模式、竞争护城河
├── 应用层：产品形态、人机协作模式、信任架构
├── 技术层：模型选型、RAG/Agent/FT 决策、推理基础设施
└── 数据层：数据资产评分、本体论设计、数据飞轮

  每层分析完后会问：要生成执行级补充文档吗？

Phase 3：文档生成
└── 战略报告（面向投资人/CEO）
└── 架构文档（含 Mermaid 图）

Phase 4：PPT 生成
└── 调用 pptx-author skill，14 张幻灯片
```

**执行级补充文档**（每层可选，面向开发/产品团队）：

| 层 | 补充文档 |
|---|---|
| 技术层 | 核心数据 Schema 草稿、Orient Prompt 模板、ERP 集成清单 |
| 应用层 | 用户故事、主流程用户旅程、通知规格说明 |

---

### 模式 C — 版本比对

**触发**：上传上一版战略报告 + 说"做第二版/对比上一版"

**做什么**：
1. 从旧文档提取**决策注册表**（所有关键架构决策）
2. 收集这段时间发生的变化（AI 能力、客户反馈、竞争、团队）
3. 逐条评估每个决策的状态：
   - ✅ 已验证
   - ❌ 已推翻
   - 🔄 已更新
   - ⏳ 仍待验证
   - 🆕 新增决策点
4. 生成 **Delta 文档**

**输出文件**：`delta-YYYY-MM-to-YYYY-MM.md`

---

## 所有输出文件一览

| 文件名 | 内容 | 受众 |
|---|---|---|
| `strategy-report-YYYY-MM.md` | 完整战略报告 | 投资人、CEO |
| `architecture-doc-YYYY-MM.md` | 架构文档（含图） | CTO、技术团队 |
| `stakeholder-<角色>-<名>.md` | 单人访谈记录 | 内部参考 |
| `stakeholder-synthesis.md` | 多方综合分析 | 架构设计输入 |
| `delta-YYYY-MM-to-YYYY-MM.md` | 版本比对 Delta | 创始团队 |
| `product-brief.md` | 产品摘要（含创始团队资产） | 内部对齐 |
| 执行级补充文档（多个） | Schema / Prompt / 用户故事等 | 开发/产品团队 |

**文件命名约定**：战略报告和 Delta 文档统一用日期（`YYYY-MM`），方便版本管理。

## 版本管理工作流

```
第一次使用
│
├── 跑完 Phase 1-4，生成 strategy-report-2026-07.md
└── 推送到 GitHub: analysis/2026-07/

     ↓ 3-6 个月后，发生了一些变化

第二次使用
│
├── 打开新对话
├── 上传 strategy-report-2026-07.md
├── 说"做第二版，对比上一版"→ 进入模式 C
├── 生成 delta-2026-07-to-2026-10.md
├── 基于 Delta 重新跑被推翻的层的分析
└── 生成 strategy-report-2026-10.md
```

**建议触发新版本的时机**：
- 重要模型发布（影响技术选型判断）
- 获得第一批真实用户反馈
- 竞争对手有重大动作
- 融资节点前
- 距上次分析超过 6 个月

## 核心概念速查

**AI 原生 vs AI 增强**：去掉 AI，产品根本不存在 = AI 原生；去掉 AI，产品变差但还存在 = AI 增强。

**Transformation Test（变革测试）**：每次分析的起点。三个问题：这打破了什么旧范式？为什么是现在？为什么巨头无法照抄？

**数据飞轮**：用户交互 → 信号捕获 → 模型改进 → 更好产品 → 更多用户。Skill 在每层分析中都会问飞轮如何设计。

**执行级 vs 战略级输出**：战略级给投资人/CEO 看（"是什么、为什么"）；执行级给开发/产品团队用（"怎么做"）。两者是独立文件，不混在一起。
